Overview

As security professionals, we spend countless hours in the terminal—conducting penetration tests, responding to incidents, analyzing malware, and managing infrastructure. Terminal multiplexers like Tmux aren’t just productivity tools; they’re essential infrastructure that can dramatically enhance your workflow, improve documentation quality, and ensure critical activities are never lost to accidental disconnections.

This comprehensive guide walks through configuring Tmux specifically for security operations, with an emphasis on automated logging, session persistence, and operational security. Whether you’re conducting a penetration test, performing incident response, or managing security infrastructure, this configuration will transform how you work in the terminal.

The Problem Tmux Solves for Security Professionals

Have you ever lost hours of work because your SSH session dropped? Needed to share exactly what commands you ran during an assessment? Wanted to document findings without breaking your workflow? Or perhaps you’ve struggled to keep track of multiple parallel tasks across different systems?

Tmux addresses all these challenges and more:

  • Session Persistence: Maintain terminal sessions across disconnections, VPN drops, or system reboots. Start a penetration test in the morning, close your laptop, and return to the exact same state hours later.

  • Activity Logging: Automatically capture every command and output with zero effort. This creates an irrefutable audit trail perfect for documentation, compliance, or demonstrating methodology in penetration testing reports.

  • History Preservation: Save and search through complete command histories from entire sessions. When you need to reference or reconstruct what you did hours or days ago, everything is available.

  • Flexible Workflows: Use multi-pane layouts to run parallel security tools side-by-side—have nmap scanning, Burp Suite intercepting, and a shell session active simultaneously.

  • Screen Capture: Generate textual screenshots of terminal output instantly. Perfect for documenting vulnerabilities, capturing evidence, or creating technical documentation.

Key Benefits for Security Work

Session Persistence: Never lose work to network issues again. Tmux sessions run in the background, detached from your terminal window. You can reconnect from anywhere and pick up exactly where you left off.

Activity Logging: Automated logging captures every keystroke and output, creating comprehensive audit trails without manual effort. This is invaluable for penetration testing documentation, incident response records, and compliance requirements.

History Preservation: Complete command histories are preserved and searchable, enabling you to quickly reference previous commands, analyze your methodology, or reproduce results.

Flexible Workflows: Multi-pane and multi-window layouts let you run parallel operations. Have your reconnaissance, exploitation, and post-exploitation activities all visible simultaneously, switching between them with keyboard shortcuts.

Screen Capture: Instant textual screenshots of terminal output eliminate the need for manual transcription. Perfect for documentation, evidence collection, and reporting.

Prerequisites

Before diving into the configuration, ensure you have the following:

Operating System Compatibility: This setup works seamlessly on Linux, macOS, and Windows (via Cygwin or WSL). The configuration and plugins are cross-platform, ensuring consistent behavior across your workstations.

Git Installation: Git is required to install Tmux Plugin Manager (TPM). If you don’t have Git installed, use your package manager:

  • Debian/Ubuntu: sudo apt install git
  • macOS: brew install git
  • Fedora/Red Hat: sudo dnf install git

Tmux Installation: Obviously, you’ll need Tmux itself. Install it via your package manager:

  • Debian/Ubuntu: sudo apt install tmux
  • macOS: brew install tmux
  • Fedora/Red Hat: sudo dnf install tmux

Terminal Emulator: While Tmux works in most terminal emulators, modern terminals like iTerm2, Alacritty, or the Windows Terminal provide better support for 256-color themes, Unicode characters, and advanced features.

Installation

The installation process is straightforward but critical to get right. Follow these steps carefully to ensure a fully functional Tmux environment optimized for security work.

1. Install Tmux Plugin Manager (TPM)

TPM is a plugin manager that simplifies installing and updating Tmux plugins. It’s maintained by the same community that develops many of the best Tmux plugins.

Execute the following command to clone the TPM repository:

git clone https://github.com/tmux-plugins/tpm ~/.tmux/plugins/tpm

This creates the TPM directory structure in your home directory, which is the standard location for Tmux plugins and configurations.

2. Configure .tmux.conf

The .tmux.conf file is Tmux’s configuration file, typically located in your home directory. If it doesn’t exist, create it. This configuration includes essential plugins and optimizations specifically chosen for security workflows.

Add the following configuration to your ~/.tmux.conf file:

# ========== PLUGINS ==========
# Core plugin manager - required for all other plugins
set -g @plugin 'tmux-plugins/tpm'

# Automated logging - captures all output for audit trails
set -g @plugin 'tmux-plugins/tmux-logging'

# Session persistence - saves and restores Tmux sessions
set -g @plugin 'tmux-plugins/tmux-resurrect'

# Enhanced navigation - fuzzy search through scrollback history
set -g @plugin 'roosta/tmux-fuzzback'

# Continuum - automatic session restoration (optional)
# Uncomment to auto-start Tmux on login
#set -g @plugin 'tmux-plugins/tmux-continuum'

# ========== CONTINUUM (OPTIONAL) ==========
# Continuum automatically saves Tmux sessions and can restore them
# when you start a new terminal. This is useful for always-on workflows.
# Uncomment the following line to auto-start Tmux on login:
#set -g @continuum-boot 'on'

# ========== HISTORY LIMIT ==========
# The scrollback buffer determines how many lines of history Tmux retains.
# Higher values mean you can capture more history, but use more memory.
# 50000 lines is a good balance for modern systems.
set -g history-limit 50000

# ========== PLUGIN MANAGER ==========
# This line initializes TPM and must be at the very bottom of your config.
# It loads all the plugins specified above.
run '~/.tmux/plugins/tpm/tpm'

Configuration Notes:

  • History Limit: The history-limit setting is critical for security work. A value of 50,000 lines ensures you can capture complete session histories for later export. Modern systems handle this easily without performance issues.

  • Continuum Plugin: This plugin (currently commented out) provides automatic session saving and restoration. It’s useful if you want Tmux sessions to persist across reboots, but may be overkill for simple workflows.

  • Plugin Selection: These four plugins (tpm, tmux-logging, tmux-resurrect, tmux-fuzzback) provide a comprehensive foundation for security work. Each serves a specific purpose and integrates seamlessly.

3. Apply Configuration

After creating or updating your .tmux.conf file, you need to reload the configuration for changes to take effect. There are two ways to do this:

From within an existing Tmux session:

tmux source ~/.tmux.conf

From outside Tmux (if you’re not in a session):

tmux new-session -d -s temp "tmux source ~/.tmux.conf; exec bash"

This reloads the configuration without disrupting your current Tmux sessions.

4. Install Plugins

Once your configuration is loaded, install the specified plugins using TPM:

  1. Ensure you’re inside a Tmux session. If not, start one: tmux
  2. Press the prefix key (default: Ctrl + b), then press Shift + I (capital ‘I’)
  3. You’ll see notifications at the bottom of the screen indicating which plugins are being installed
  4. Wait for the “done” notification, indicating all plugins are installed

Understanding the Prefix: The Tmux prefix key (default Ctrl + b) is your gateway to all Tmux commands. Press it first, then press the command key. Think of it like a mode switch—you press the prefix to “enter command mode,” then press the specific command key.

Troubleshooting Installation:

  • If plugins don’t install, verify your internet connection and Git installation
  • Check that the ~/.tmux/plugins/tpm directory exists
  • Ensure the configuration file is properly formatted (no syntax errors)
  • If all else fails, manually clone each plugin into ~/.tmux/plugins/

Logging Capabilities

The tmux-logging plugin is a game-changer for security professionals. It provides multiple logging modes, each optimized for different security workflows. Whether you need real-time logging, quick captures, or complete history exports, tmux-logging has you covered.

Mode 1: Real-Time Activity Logging

Purpose: Real-time logging captures everything you type and everything the terminal outputs as it happens. This is your primary logging mode for most security operations, creating a comprehensive, timestamped record of all activities.

How It Works: When activated, tmux-logging intercepts all terminal I/O (input/output) for the current pane and writes it to a log file. This includes commands, program output, error messages, and any other text that appears in your terminal.

Activation: Prefix + Shift + p

Toggle real-time logging on or off for the current pane. When you activate it, you’ll see an indicator in your status bar confirming logging is active.

File Format:

tmux-#{session_name}-#{window_index}-#{pane_index}-%Y%m%dT%H%M%S.log

This format ensures unique filenames and provides context about which session, window, and pane generated the log.

Real-World Example:

~/tmux-pentest-external-0-1-20250527T165614.log

From this filename, you can immediately identify:

  • Session name: pentest-external
  • Window: 0
  • Pane: 1
  • Timestamp: 2025-05-27 at 16:56:14

Location: $HOME (your user home directory)

When to Use Real-Time Logging:

Penetration Testing Activities: Activate logging before starting any penetration test engagement. This captures the entire methodology from reconnaissance through exploitation and post-exploitation, providing an irrefutable record of your activities for client reports and compliance documentation.

Incident Response: During incident response activities, real-time logging creates a chronological record of your investigation steps. This is invaluable for post-incident analysis, stakeholder communication, and legal documentation if required.

Command Auditing: For security-critical activities where every command matters, real-time logging ensures nothing is missed. Use it when executing destructive commands, making configuration changes, or performing privilege escalation attempts.

Methodology Documentation: When building or testing new attack techniques, real-time logging captures your experimentation process. You can later review exactly what commands you tried, in what order, and what the results were—perfect for creating detailed methodology documentation.

Training and Knowledge Transfer: During training or when mentoring junior security professionals, real-time logging captures complete sessions that can be reviewed later to understand thought processes, command usage, and workflow patterns.

Mode 2: Screen Capture

Purpose: Screen capture saves the currently visible text in your terminal as a text file. Think of it as taking a screenshot, but instead of an image, you get searchable, copy-pasteable text. This is invaluable for quickly capturing findings or tool output without starting a full logging session.

How It Works: Tmux-logging captures the text currently displayed in your terminal viewport and saves it to a file. Unlike real-time logging, which captures continuous output, screen capture is a one-time snapshot of what’s visible at that exact moment.

Activation: Prefix + Alt + p

Instantly capture the visible text and save it to a log file. The capture is instantaneous—no waiting or indication in the status bar.

File Format:

tmux-screen-capture-#{session_name}-#{window_index}-#{pane_index}-%Y%m%dT%H%M%S.log

Real-World Example:

~/tmux-screen-capture-forensics-2-0-20250527T173022.log

Location: $HOME (your user home directory)

When to Use Screen Capture:

Documenting Vulnerability Findings: When you discover a vulnerability, quickly capture the exploit output, error messages, or tool results. This provides evidence for your penetration testing reports and eliminates manual transcription.

Capturing Tool Output: Many security tools (like nmap, sqlmap, or Metasploit) output lengthy results. Use screen capture to save specific findings without starting a logging session or selecting text manually.

Report Evidence Collection: When building penetration testing reports or incident response documentation, screen captures provide textual evidence that can be directly pasted into reports. This is cleaner than screenshots and fully searchable.

Quick Documentation: Sometimes you need to document something quickly without disrupting your workflow. Screen capture is perfect for these moments—press a key, capture the output, and continue working.

Evidence for Findings: In penetration testing, showing exactly what a tool returned is crucial. Screen captures of SQL injection output, buffer overflow results, or privilege escalation output provide irrefutable evidence.

Mode 3: Complete History Export

Purpose: Complete history export saves the entire scrollback buffer from the beginning of the pane session. This is your retroactive logging option—perfect when you forgot to start real-time logging but still need to capture everything that happened in the session.

How It Works: Tmux maintains a scrollback buffer of all previous output (determined by history-limit). The complete history export saves this entire buffer to a file, creating a comprehensive log of everything from the moment the pane was created up to the current point.

Activation: Prefix + Alt + Shift + p

Export the complete scrollback buffer for the current pane to a log file.

File Format:

tmux-history-#{session_name}-#{window_index}-#{pane_index}-%Y%m%dT%H%M%S.log

Real-World Example:

~/tmux-history-malware-analysis-1-2-20250527T184533.log

Location: $HOME (your user home directory)

Critical Dependency: This feature depends entirely on your history-limit configuration. Only lines within the scrollback buffer will be saved—if your history limit is too low, you’ll lose older output. The recommended setting of 50,000 lines provides ample buffer for most sessions.

When to Use Complete History Export:

Retroactive Logging: We’ve all been there—you’re hours into an engagement, realize you forgot to start logging, and need to capture what you’ve done. Complete history export saves the day, capturing everything up to this point.

Session Documentation: Before closing a long-running session, export the complete history for documentation purposes. This ensures you have a complete record even if you didn’t use real-time logging.

Post-Assessment Review: After completing a penetration test or investigation, export complete histories for each pane to create comprehensive documentation for reports and debriefings.

Comprehensive Audit Trails: For engagements requiring complete documentation, export histories for all sessions before cleanup. This creates a complete record for legal, compliance, or client requirements.

Methodology Analysis: Reviewing complete history exports helps identify patterns in your methodology, understand what approaches worked (and didn’t), and improve your security practices.

Mode 4: Clear Pane History

Purpose: Clearing pane history removes all scrollback content from the buffer. This is essential for operational security—removing sensitive information, resetting session state, or maintaining clean working environments.

How It Works: Clearing history doesn’t affect your running processes or current session—it simply removes the scrollback buffer that stores previous output. New commands and output will start a fresh buffer.

Activation: Prefix + Alt + c

Immediately clear the scrollback buffer for the current pane.

When to Clear Pane History:

Removing Sensitive Data: After capturing passwords, API keys, or other sensitive information in terminal output, clear the history to prevent accidental exposure. Even though logs might be saved elsewhere, clearing scrollback prevents shoulder surfing or accidental disclosure.

Resetting Session State: When starting a new phase of an engagement, clearing history provides a fresh start. This is especially useful when transitioning from reconnaissance to exploitation phases.

Reducing Memory Footprint: For very long sessions with extensive output, clearing history periodically reduces memory usage. While modern systems handle large buffers well, this can improve performance on resource-constrained systems.

Operational Security: Before leaving your workstation or sharing your screen, clear scrollback to remove sensitive information that might be visible. This is a simple but effective OpSec practice.

Avoiding Confusion: When working on multiple engagements in the same session, clearing history between phases prevents accidentally mixing commands or output from different contexts.

Search and Navigation

Tmux’s copy mode provides powerful search and navigation capabilities for working with your scrollback history. Instead of scrolling endlessly or trying to remember exactly where you saw something, use copy mode to quickly find and review previous output.

Copy Mode

Copy mode is Tmux’s terminal-independent way of navigating and interacting with your scrollback buffer. It provides vim-like navigation, powerful search capabilities, and the ability to copy text from your history.

Why Copy Mode Matters: Modern terminals have their own scrolling and selection capabilities, but they’re inconsistent across platforms and often limited in functionality. Copy mode provides a consistent, powerful interface that works identically on Linux, macOS, and Windows.

Enter Copy Mode: Ctrl + a + [

This activates copy mode for the current pane. The mode indicator in your status bar changes, and you can now navigate through your scrollback history.

Navigation in Copy Mode:

Once in copy mode, use these keys to navigate:

  • Arrow Keys: Move cursor up/down/left/right
  • Page Up / Page Down: Move by pages
  • Home / End: Jump to beginning or end of buffer
  • Ctrl + u: Scroll up half a page
  • Ctrl + d: Scroll down half a page
  • Space: Start text selection (use movement keys to extend selection)
  • Enter: Copy selected text to system clipboard

Search Functionality:

Search is where copy mode really shines—quickly find commands, output, or patterns in your history:

  1. Enter Search Mode: Press / to start searching forward (or ? to search backward)
  2. Type Your Search Term: Enter the text you want to find (supports regular expressions)
  3. Navigate Results: Press n to go to the next match, Shift + n for previous match
  4. Exit Search: Continue navigation or exit copy mode

Real-World Search Examples:

Find Specific Commands: Search for command names to see when and how you used them.

  • Example: Search /nmap to find all nmap commands and their output

Locate Error Messages: Quickly find error messages to troubleshoot issues.

  • Example: Search /ERROR or /failed to find problematic commands

Review IP Addresses: Search for IP addresses to see all interactions with specific hosts.

  • Example: Search /192.168.1.100 to review all traffic to that IP

Find Vulnerability Output: Locate specific vulnerability or exploit output in lengthy output.

  • Example: Search /CVE-2023-1234 or /RCE to find relevant findings

Review Reconnaissance Data: Search through enumeration output for specific services or ports.

  • Example: Search /open port/ or /SSH to find relevant reconnaissance data

Exit Copy Mode: Press Esc twice to return to normal terminal mode

Pro Tip: If you’re in copy mode and want to run a command you found, you can:

  1. Highlight the command text with space selection
  2. Press Enter to copy it
  3. Exit copy mode (Esc twice)
  4. Press Ctrl + a then ] to paste into the command line
  5. Execute the command

Security Considerations

While Tmux logging and session management offer tremendous benefits for security workflows, they also introduce considerations around data protection and operational security. This section covers best practices for maintaining security while leveraging Tmux capabilities.

Data Protection

1. Sensitive Information in Logs

Log files generated by tmux-logging may contain highly sensitive information that needs protection:

Types of Sensitive Data:

  • Passwords and credentials (both in plaintext and potentially hashed)
  • API keys and tokens
  • IP addresses, hostnames, and network topology information
  • Confidential client data or proprietary information
  • Command-line arguments containing sensitive parameters
  • Output from tools that may reveal system configurations or vulnerabilities

Protection Strategies:

  • Review Before Sharing: Before sharing logs with colleagues or clients, review them to ensure no sensitive information is exposed. Use tools like grep to search for common sensitive patterns:

    grep -i "password\|key\|secret\|token" ~/tmux-*.log

  • Encrypt Logs at Rest: Store logs in encrypted directories or filesystems. On Linux/macOS, you can use encrypted volumes or directories (e.g., gocryptfs, encfs, or macOS FileVault). This ensures logs remain protected even if your system is compromised.

  • Secure Deletion: After retention periods expire or when logs are no longer needed, delete them securely. On Linux: shred -u filename, on macOS: rm -P filename. Don’t just use rm—recovering deleted files is trivial.

  • Log Sanitization: Before sharing logs, consider redacting or anonymizing sensitive information. Tools like sed or specialized log sanitizers can help:

    sed 's/password: .*/password: [REDACTED]/g' logfile.log > sanitized.log

  • Access Control: Set restrictive permissions on log files (typically chmod 600) so only the owner can read or write. Avoid world-readable permissions (chmod 644 or 777).

2. Log File Location Management

By default, tmux-logging saves files to your home directory ($HOME), which may not be the optimal location for security-sensitive logs.

Alternative Locations:

  • Encrypted Directory: Create an encrypted directory for logs:

    mkdir -p ~/secure-logs
# Configure tmux-logging to use this directory (if supported)

  • Project-Specific Directories: Organize logs by engagement or project:

    mkdir -p ~/security-logs/engagement-name
# Move logs here after each session

  • Temporary Directory: Use /tmp or a temporary filesystem for logs that don’t need persistence, ensuring automatic cleanup on reboot.

Log Rotation Policies: Implement log rotation to prevent unbounded disk usage:

  • Archive old logs to compressed files
  • Delete logs after defined retention periods
  • Monitor disk usage of log directories
  • Automate cleanup with cron jobs or systemd timers

3. History Limit Configuration

The history-limit setting determines how much output Tmux retains in memory. This setting directly impacts logging capabilities and resource usage.

Trade-offs:

  • Higher Limits (e.g., 100,000+):

    • Pros: Capture more history, better for retroactive logging
    • Cons: Higher memory usage per pane, slower history navigation
    • Use case: Long-running engagements, complex analysis tasks

  • Moderate Limits (e.g., 10,000-50,000):

    • Pros: Good balance of history and memory usage
    • Cons: May lose very old output in very long sessions
    • Use case: Most penetration testing and incident response work

  • Lower Limits (e.g., 1,000-5,000):

    • Pros: Minimal memory overhead, fast navigation
    • Cons: Limited retroactive logging capability
    • Use case: Short sessions, resource-constrained systems

Recommendation: Start with 50,000 lines (as configured). Monitor memory usage during typical sessions and adjust based on your needs and system capabilities.

Operational Security

1. Session Isolation and Naming

Proper session management prevents cross-contamination of logs and improves organization.

Descriptive Session Names: Create named sessions for each engagement or task:

tmux new-session -s client-xyz-pentest
tmux new-session -s incident-respond-2025-05-27
tmux new-session -s malware-analysis-suspect-sample

Benefits of Isolated Sessions:

  • Clear Log Separation: Each session’s logs are distinctly labeled, preventing confusion
  • Context Preservation: Switch between engagements without mixing commands or output
  • Clean Workflow: Close and reopen sessions to create clean states for new tasks
  • Audit Trail Clarity: Session names provide immediate context in log filenames

2. Regular Backups of Critical Logs

For long-running or critical engagements, ensure log files are backed up regularly.

Backup Strategies:

  • Automated Sync: Use rsync to sync logs to backup storage:

    rsync -avz ~/tmux-*.log backup-server:/backups/security-logs/

  • Cloud Storage with Encryption: Upload to cloud storage (AWS S3, Azure Blob, etc.) with encryption enabled. Use tools like rclone for automation.

  • Version Control for Methodologies: Consider committing non-sensitive log files or methodology summaries to version control for future reference and knowledge sharing.

  • Periodic Exports: Use complete history export at defined intervals (e.g., hourly, after each phase) to create incremental backups of session progress.

Encryption During Transfer: Always encrypt logs during transmission:

# Using scp with SSH (already encrypted)
scp ~/tmux-*.log remote:/secure-location/

# Using rclone with encrypted destinations
rclone copy ~/tmux-*.log encrypted-remote:logs/

3. Access Control and File Permissions

Ensure only authorized personnel can access your Tmux logs.

File Permissions:

# Restrict to owner only
chmod 600 ~/tmux-*.log

# Verify permissions
ls -l ~/tmux-*.log

Directory Permissions:

# Restrict log directory to owner
chmod 700 ~/secure-logs/

# Set default permissions for new files in directory
chmod g+s ~/secure-logs/  # (if using group)

System-level Controls: For shared systems, consider:

  • Access Control Lists (ACLs) for fine-grained permissions
  • Mandatory Access Control (MAC) systems (SELinux, AppArmor) for additional protection
  • User groups to restrict log access to specific teams

Principle of Least Privilege: Only grant log access to those who absolutely need it. In team environments, create specific log access roles and audit log access attempts.

File Naming Convention

All log files generated by tmux-logging follow this standardized naming format:

tmux-[type]-[session_name]-[window_index]-[pane_index]-[timestamp].log

Component Breakdown:

Component Description Example
type Logging mode type screen-capture, history, or omitted for real-time
session_name Name of the Tmux session pentest-external, incident-response
window_index Window number (0-based) 0, 1, 2, etc.
pane_index Pane number within window (0-based) 0, 1, 2, etc.
timestamp ISO 8601 format timestamp 20250527T165614 = May 27, 2025 at 16:56:14

Example Filenames Explained:

tmux-pentest-external-0-1-20250527T165614.log
  • Real-time logging (no type prefix)
  • Session: pentest-external
  • Window 0, Pane 1
  • Captured on May 27, 2025 at 16:56:14
tmux-screen-capture-forensics-2-0-20250527T173022.log
  • Screen capture
  • Session: forensics
  • Window 2, Pane 0
  • Captured on May 27, 2025 at 17:30:22
tmux-history-malware-analysis-1-2-20250527T184533.log
  • Complete history export
  • Session: malware-analysis
  • Window 1, Pane 2
  • Captured on May 27, 2025 at 18:45:33

Naming Best Practices:

  • Use descriptive session names that provide context
  • The timestamp format ensures chronological sorting
  • Window and pane indices help reconstruct session layout
  • Type prefix indicates logging mode at a glance

Best Practices

Follow these proven practices to maximize the effectiveness of your Tmux workflow for security operations.

1. Always Start Logging Early

The Golden Rule: Activate real-time logging (Prefix + Shift + p) at the very beginning of every engagement or important session, before you run any commands.

Why This Matters:

  • Complete Audit Trail: Starting early ensures you capture everything from the first command, creating a complete record for documentation and compliance
  • No Missed Context: If you forget to log initially, you’ll miss reconnaissance, enumeration, or other early-phase activities that are crucial for methodology documentation
  • Peace of Mind: Knowing everything is logged allows you to focus on the engagement without worrying about documentation

When to Activate Logging:

  • Before starting penetration testing assessments
  • When beginning incident response investigations
  • Prior to malware analysis sessions
  • Before executing destructive or irreversible commands
  • When starting new security research or experimentation

Pro Tip: Make it muscle memory—the moment you start a new engagement or task, your first action should be activating logging.

3. Regular Screen Captures for Documentation

Take periodic screen captures (Prefix + Alt + p) throughout your engagements to document findings, key outputs, and evidence.

When to Capture Screens:

  • Vulnerability Discovery: Immediately after discovering a vulnerability, capture the exploit output or evidence
  • Tool Output: Save results from enumeration tools (nmap, enum4linux, etc.) for easy reference
  • Error Messages: Capture error messages for troubleshooting and documentation
  • Configuration Changes: Document system configurations before and after changes
  • Testing Results: Save output from successful exploit attempts or privilege escalations

Screen Capture Workflow:

  1. Run your security tool or command
  2. Review the output to confirm it’s complete
  3. Press Prefix + Alt + p to capture
  4. Continue with your workflow (no interruption)

Benefits Over Manual Copying:

  • Faster than selecting and copying text
  • Captures complete output (not truncated by viewport)
  • Preserves formatting and structure
  • Creates timestamped files automatically

4. Export History Periodically

Use complete history exports (Prefix + Alt + Shift + p) at key points in your engagement to create comprehensive session backups and documentation.

When to Export Histories:

  • Before Clearing History: Always export before clearing scrollback buffers
  • At Phase Transitions: Export when moving between engagement phases (reconnaissance → exploitation)

  • After Complex Tasks: Export after completing complex, multi-step procedures

  2. Press Prefix + Alt + Shift + p
  3. Wait for the file to be created (check status bar notification)
  4. Optionally rename or move the file for better organization
  5. Continue working or proceed to next task

5. Secure Log Storage

Store your Tmux logs securely to protect sensitive information and maintain operational security.

Secure Storage Practices:

Encrypted Directories:

# Create encrypted directory (using gocryptfs as example)
mkdir -p ~/encrypted-logs
gocryptfs -init ~/encrypted-logs
gocryptfs ~/encrypted-logs ~/secure-logs

# Configure tmux-logging to use ~/secure-logs/

Appropriate Permissions:

# Restrict to owner only
chmod 600 ~/tmux-*.log

# Ensure log directory is restricted
chmod 700 ~/secure-logs/

Regular Cleanup:

# Remove old logs (example: older than 90 days)
find ~/secure-logs/ -name "tmux-*.log" -mtime +90 -delete

# Or compress old logs
find ~/secure-logs/ -name "tmux-*.log" -mtime +30 -exec gzip {} \;

Backup Strategy:

# Sync to secure backup location
rsync -avz --delete ~/secure-logs/ /backup/encrypted-storage/security-logs/

# Or upload to encrypted cloud storage
rclone sync ~/secure-logs/ encrypted-remote:security-logs/

Troubleshooting

Even the most well-configured systems encounter issues. This section covers common Tmux problems and their solutions.

Plugins Not Loading

Symptoms: Tmux starts, but plugin functionality (logging, resurrect, etc.) doesn’t work.

Troubleshooting Steps:

  1. Verify TPM Installation:

    # Check if TPM directory exists
ls -la ~/.tmux/plugins/tpm

# Expected output should show directory contents

  2. Reload Tmux Configuration:

    # From outside Tmux
tmux source ~/.tmux.conf

# Or from within Tmux session
tmux source-file ~/.tmux.conf

  3. Reinstall Plugins:

    # In a Tmux session, press:
Prefix + I
# Wait for "done" notification

  4. Check Configuration File:

    # Verify config file exists
ls -la ~/.tmux.conf

# Check for syntax errors
cat ~/.tmux.conf | grep -i error

  5. Manually Clone Plugins (if automated install fails):

    cd ~/.tmux/plugins/
git clone https://github.com/tmux-plugins/tmux-logging
git clone https://github.com/tmux-plugins/tmux-resurrect
git clone https://github.com/roosta/tmux-fuzzback

Common Causes:

  • Network issues preventing git clone operations
  • Syntax errors in .tmux.conf file
  • Missing dependencies or Git not installed
  • File permissions preventing plugin installation

Logs Not Appearing

Symptoms: You activate logging, but no log files appear in your home directory.

Troubleshooting Steps:

  1. Check Home Directory Permissions:

    # Verify you can write to home directory
ls -la ~

# Test file creation
touch ~/test-write && rm ~/test-write

  2. Verify File System Has Space:

    # Check disk usage
df -h ~

# If disk is full, clear space or use alternative location

  3. Confirm Logging is Active:

    • Look for logging indicator in Tmux status bar (usually shows “LOGGING ON”)
    • Check if status bar is visible (Prefix + : then set status on)

  4. Check Tmux Version:

    tmux -V

# Ensure version is 2.1 or higher for full logging support

  5. Manually Test Plugin:

    # In Tmux, check if plugin is loaded
Prefix + :
# Run command: list-keys -T root
# Look for logging-related key bindings

Common Causes:

  • Full disk partition
  • Incorrect file permissions
  • Tmux version too old (upgrade required)
  • Plugin not properly installed or loaded
  • Status bar disabled, hiding logging indicator

Search Not Working in Copy Mode

Symptoms: You enter copy mode but search (/) doesn’t find expected text.

Troubleshooting Steps:

  1. Ensure Copy Mode is Active:

    • Look for visual indicator in status bar (usually shows “COPY MODE”)
    • If not active, press Ctrl + a + [ to enter

  2. Verify Search Syntax:

    • Ensure you’re using forward slash / for forward search
    • Check for special characters that need escaping
    • Try simpler search terms first (e.g., search for “ERROR” instead of complex regex)

  3. Check History Limit:

    # Verify history limit in config
grep history-limit ~/.tmux.conf

# If too low, content may have aged out of buffer
set -g history-limit 50000  # Add or increase this line

  4. Test with Known Content:

    • Type a unique string like “TEST12345” in the terminal
    • Enter copy mode and search for it
    • If found, search is working; issue was search term didn’t exist

  5. Check Scrollback Buffer:

    # In Tmux, check how many lines in buffer
Prefix + :
# Run command: display-message 'scrollback: #{scrollback_history_size}'

# If 0 or very low, history may be cleared or not capturing

Common Causes:

  • Copy mode not actually active
  • Search term doesn’t exist in history
  • History was cleared or aged out
  • Complex search patterns not matching
  • Buffer cleared recently, removing searchable content
